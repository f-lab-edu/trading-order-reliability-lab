# 1. 프로젝트 문제 정의

## 1.1 프로젝트명

**해외주식 주문 상태 가시성 및 정합성 복구 플랫폼**

### 부제

외부 브로커 응답 불확실성에 대응하는 주문 상태 추적 시스템

---

## 1.2 프로젝트 배경

해외주식 주문 처리 흐름은 단순한 동기식 CRUD가 아니다.

사용자가 주문을 요청하면 내부 주문 시스템은 외부 브로커 또는 대외기관으로 주문을 전달하고, 이후 주문 접수, 거절, 부분체결, 완전체결, 취소, 만료와 같은 이벤트를 비동기적으로 수신한다.

이 과정에서 다음과 같은 문제가 발생할 수 있다.

* 외부 브로커 응답 지연
* 주문 접수 ACK 유실
* 브로커 응답 중복 수신
* 체결 이벤트 중복 수신
* ACK보다 체결 이벤트가 먼저 도착하는 순서 역전
* 부분체결 이후 취소 요청이 들어오는 경합
* 취소 요청 중 추가 체결 발생
* DAY 주문의 장 마감 만료
* 내부 주문 상태와 외부 브로커 상태의 일시적 불일치
* 사용자가 보는 주문 상태와 실제 브로커 상태의 괴리

따라서 이 프로젝트는 단순히 주문 생성 API를 구현하는 것이 아니라, **불완전하고 비동기적인 외부 응답 환경에서도 사용자가 신뢰할 수 있는 주문 상태를 제공하고, 애매한 상태를 최종적으로 수렴시키는 구조**를 설계·구현하는 것을 목표로 한다.

---

## 1.3 핵심 문제 정의

이 프로젝트가 해결하려는 핵심 문제는 다음과 같다.

> 해외주식 주문이 외부 브로커로 전달되는 과정에서 응답 지연, 유실, 중복, 순서 꼬임, 부분체결/취소 경합이 발생하더라도 사용자가 현재 주문 상태를 신뢰할 수 있도록 추적하고, 내부 상태와 브로커 상태가 불일치할 경우 재조회와 복구 절차를 통해 최종 상태로 수렴시키는 시스템을 설계·구현한다.

요약하면 다음과 같다.

> **외부 브로커 연계의 불확실성 속에서 해외주식 주문 상태를 실시간으로 가시화하고, 불일치 상태를 복구 가능한 구조로 만드는 것**

---

## 1.4 사용자 관점의 문제

이 프로젝트의 1차 사용자는 해외주식 주문을 수행하는 투자자다.

투자자는 주문 요청 이후 다음과 같은 불확실성을 겪을 수 있다.

* 주문이 실제로 접수되었는지 알 수 없다.
* 주문이 거절되었는지, 아직 확인 중인지 구분하기 어렵다.
* 일부만 체결되었는지 전량 체결되었는지 즉시 알기 어렵다.
* 취소 요청이 접수된 것인지, 취소가 확정된 것인지, 취소 중 체결된 것인지 알기 어렵다.
* 장 마감 이후 미체결 잔량이 어떻게 처리되었는지 명확하지 않다.
* 장애 상황에서 주문 상태가 실패인지, 확인 중인지, 복구 중인지 구분되지 않으면 불안하다.

따라서 사용자-facing 목표는 다음과 같다.

> 사용자가 주문 상세 화면 또는 주문 목록에서 현재 주문 상태, 누적 체결 수량, 잔여 수량, 취소 여부, 만료 여부, 확인 중 상태를 신뢰할 수 있게 제공한다.

---

## 1.5 운영 관점의 문제

이 프로젝트의 2차 사용자는 운영자 또는 서버 개발자다.

운영 관점에서는 다음 문제가 중요하다.

* 특정 주문이 왜 `UNKNOWN` 상태가 되었는지 추적하기 어렵다.
* 브로커로 전문이 실제 전송되었는지, 응답이 왔는지, 파싱에 실패했는지 확인하기 어렵다.
* 중복 체결 이벤트를 재처리했을 때 주문 상태가 오염될 수 있다.
* 브로커 상태와 내부 주문 상태가 다를 때 체계적인 수렴 절차가 없으면 수동 DB 보정에 의존하게 된다.
* 장애 재현이 어렵다면 테스트가 정상 흐름 위주로만 끝난다.

따라서 운영-facing 목표는 다음과 같다.

> 주문 이벤트 타임라인, 브로커 전문 송수신 기록, command attempt 이력, reconciliation job 이력을 통해 주문 상태 변경 원인을 추적하고, 애매한 주문 상태를 재조회 기반으로 복구할 수 있게 한다.

## 1.6 1차 프로젝트 범위

### 포함 범위

1차 구현에서는 다음 범위를 다룬다.

#### 주문 도메인

* 해외주식 지정가 주문
* `TIF = DAY` 고정
* 주문 생성
* 주문 조회
* 주문 취소
* 주문 상태 실시간 알림
* 주문 접수
* 주문 거절
* 부분체결
* 완전체결
* 부분체결 후 취소
* 취소 거절
* 장 마감에 따른 미체결 잔량 만료 `EXPIRED`

#### 시장 상태

* 시장 상태는 1차 범위에서 `OPEN`, `CLOSED`만 지원한다.
* 정교한 장전, 정규장, 장후, 휴장 캘린더는 1차 범위에서 제외한다.

#### 외부 브로커 연계

* 실제 브로커 대신 Netty 기반 TCP broker mock을 사용한다.
* broker mock은 stateful broker simulator로 동작한다.
* 실제 금융권 전문 통신 구조를 단순화하여 다음을 구현한다.

  * TCP length-prefixed frame
  * 공통 전문 header
  * 고정 길이 body
  * 전문 ID별 parser/serializer
  * 주문 접수/거절/체결/취소/만료/상태조회 전문
  * field padding
  * numeric zero padding
  * wireMessageId/traceId 기반 correlation
  * malformed 전문 처리

#### 비동기 처리 및 정합성

* Kafka 기반 비동기 이벤트 전달
* DB 기반 주문 상태 저장
* Outbox/Inbox 기반 메시지 신뢰성 보강
* `UNKNOWN` 상태와 reconciliation 기반 복구
* 중복 이벤트 처리
* out-of-order 이벤트 처리
* ACK 유실/timeout 처리

#### 사용자 인터페이스

* REST API
* SSE 기반 주문 상태 실시간 알림

#### 운영/관측성

* 주문 이벤트 이력
* 브로커 전문 송수신 journal
* command attempt 이력
* reconciliation job 이력
* 기본 메트릭과 로그

## 1.8 심화 범위

1차 구현과 테스트가 안정화된 이후 다음 순서로 확장한다.

### 심화 1: 멀티 브로커 라우팅 / fallback

* 브로커 A/B 추가
* 브로커별 장애 감지
* 라우팅 정책 구현
* primary broker 장애 시 fallback 처리
* 브로커별 ACK 지연 측정
* 브로커별 실패율 측정
* 브로커별 상태 비교

### 심화 2: 운영 콘솔

* 주문 이벤트 타임라인 조회
* 브로커 전문 송수신 이력 조회
* command attempt 조회
* reconciliation job 조회
* UNKNOWN 주문 목록 조회
* 수동 reconciliation 트리거
* DLQ 또는 재처리 대상 이벤트 조회

### 후순위 검토: FIX 기반 브로커 어댑터

FIX는 프로젝트의 핵심 범위가 아니라 브로커 통신 어댑터의 한 구현체로 본다.

따라서 FIX는 1차 및 주요 심화 범위에서 제외하고, 프로젝트가 충분히 안정화된 이후 후순위로 검토한다.

## 1.9 해결 방식의 핵심 원칙

### 1. 주문 상태의 진실은 Order Service DB에 둔다

Kafka나 Broker Gateway가 주문의 최종 상태를 소유하지 않는다.
주문 상태 변경은 Order Service만 수행한다.

### 2. 외부 브로커 응답은 canonical event로 정규화한다

브로커 전문 구조를 내부 주문 도메인에 직접 노출하지 않는다.
Broker Gateway가 전문을 파싱한 뒤 내부 표준 이벤트로 변환한다.

### 3. 외부 응답은 중복·지연·순서 역전될 수 있다고 가정한다

정상 순서만 가정하지 않는다.
상태머신은 중복 이벤트와 out-of-order 이벤트를 견딜 수 있어야 한다.

### 4. 애매한 상태는 실패가 아니라 `UNKNOWN`으로 둔다

브로커에 주문이 전달되었는지 불확실한 상황에서 곧바로 실패 처리하거나 재전송하지 않는다.
`UNKNOWN` 상태로 두고 상태조회 기반 reconciliation으로 수렴시킨다.

### 5. DB 트랜잭션과 Kafka 발행/소비 사이의 장애는 Outbox/Inbox로 제어한다

분산 트랜잭션 대신 서비스별 로컬 DB에 Outbox/Inbox를 두고, at-least-once delivery와 idempotent processing을 조합해 정합성을 확보한다.

### 6. 브로커 전문 통신은 Gateway 계층에 격리한다

TCP 전문 parser/serializer는 Broker Gateway와 Broker Mock 내부에만 존재한다.
Order Service 도메인 모델은 전문 포맷에 의존하지 않는다.

## 1.10 프로젝트 성공 기준

### 기능적 성공 기준

* 동일 `clientOrderId`로 중복 주문 요청이 들어와도 주문은 한 건만 생성된다.
* 주문은 `PENDING_ACK → LIVE → PARTIALLY_FILLED → FILLED` 흐름을 정상 처리한다.
* 주문 거절 이벤트를 수신하면 `REJECTED`로 종결된다.
* 부분체결 후 취소 요청을 처리할 수 있다.
* 취소 요청 중 추가 체결이 발생해도 최종 상태가 일관되게 수렴한다.
* DAY 주문의 미체결 잔량은 장 마감 시 `EXPIRED`로 종결된다.
* 브로커 이벤트 중복 수신 시 누적 체결 수량이 중복 반영되지 않는다.
* ACK보다 체결 이벤트가 먼저 도착해도 주문 상태가 올바르게 수렴한다.
* ACK 유실 또는 timeout 발생 시 주문은 `UNKNOWN`으로 전환된다.
* `UNKNOWN` 주문은 상태조회 기반 reconciliation을 통해 최종 상태로 수렴한다.

---

### 운영적 성공 기준

* 주문별 상태 변경 이력을 이벤트 타임라인으로 확인할 수 있다.
* 브로커 전문 송수신 이력을 wireMessageId, traceId, clientOrderId, brokerOrderId 기준으로 추적할 수 있다.
* command attempt 이력을 통해 브로커 전송 성공, 실패, timeout을 구분할 수 있다.
* malformed 전문 처리 결과를 추적할 수 있다.
* Kafka 메시지 중복 소비가 주문 상태 오염으로 이어지지 않는다.
* reconciliation job 이력을 통해 복구 과정과 결과를 추적할 수 있다.

---

### 기술적 성공 기준

* Order Service, Broker Gateway, Recovery Service의 책임이 분리되어 있다.
* 주문 상태 변경 주체는 Order Service로 제한된다.
* 브로커 통신 방식이 바뀌어도 Order Service의 도메인 모델은 변경되지 않는다.
* Kafka는 상태 저장소가 아니라 이벤트 전달 수단으로 사용된다.
* DB 트랜잭션과 Kafka 발행 사이의 장애는 Outbox로 제어된다.
* Kafka 중복 소비는 Inbox로 제어된다.
* Netty 기반 broker mock은 실제 금융권 전문 통신의 핵심 구조를 단순화해 재현한다.
* 장애/중복/순서 역전/무응답 시나리오를 테스트로 재현할 수 있다.

## 1.11 프로젝트가 보여주려는 역량

이 프로젝트는 단순 주문 CRUD나 주식앱 클론이 아니다.
다음 역량을 보여주는 것을 목표로 한다.

* Java/Spring 기반 서버 애플리케이션 설계
* 금융성 도메인의 상태 모델링
* 외부 시스템 연계의 불확실성 처리
* TCP 전문 기반 대외 연계 구조 이해
* Kafka 기반 비동기 이벤트 처리
* DB 트랜잭션 중심의 정합성 설계
* Outbox/Inbox 기반 메시징 신뢰성 보강
* idempotency 설계
* 이벤트 중복 처리
* 이벤트 순서 역전 처리
* 장애 후 reconciliation 설계
* 운영 추적 가능성을 고려한 로그/저널/이력 설계
* 단계적 확장을 고려한 아키텍처 설계
